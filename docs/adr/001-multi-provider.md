# ADR 001: provider는 코드가 아니라 설정이다

- Status: Draft (최초 작성)
- Date: 2026-07-03
- 부모: [000](000-identity.md) 공리 1 (물리는 filegate가 소유한다)

## 문제

벤더마다 기능·주소·인증이 다르고, 벤더 자체도 변한다(MinIO OSS 아카이브가 설계 도중 실제로 일어났다). 벤더를 코드에 박으면 그 벤더의 운명이 filegate의 운명이 된다. 그리고 "어떤 파일이 어디로 가나"라는 지식은 서비스에 살 수 없다(공리 1).

## 결정

- **provider = 접근 계약.** 벤더 + 계정 + 주소 + 자격증명 + 공간의 묶음 하나. 같은 벤더의 다른 계정도 서로 다른 provider다 — 계정 분산·리전 분리가 이 정의에서 공짜로 나온다.
- **S3 호환 와이어가 1차 계약.** adapter 하나로 S3/R2/OCI/MinIO/Garage를 커버한다. 벤더 네이티브 adapter는 필요가 증명될 때만.
- **내부/공개 접근 주소를 구분한다.** 서명된 접근은 주소에 묶이므로, filegate가 보는 주소와 클라이언트가 접근하는 주소는 처음부터 별개 개념이다.
- **capability는 선언식.** 런타임 탐지 없음. 운영자가 선언하고, 틀리면 해당 provider에서 실패로 드러난다.
- **file(정체성)과 location(위치)을 분리한다.** 위치는 파일의 생애 동안 바뀔 수 있는 포인터다.
- **배치는 인터페이스 뒤.** 대상은 후보 풀이고, 선택 전략은 설정이다(기본: capacity 여유가 있는 후보 중 무작위). location이 가변 포인터이므로 어떤 전략을 골라도 나중에 리밸런싱(이동)으로 교정 가능하다.

## 경계선

- 자동 비용 최적화·자동 이동은 요청 경로에 넣지 않는다 — reconciler의 관할.
- 객체 이름은 filegate가 발급한 불투명 키다. 클라이언트 파일명이 아니다.
- 비용·무료 구간은 변하는 외부 사실 — ADR이 아니라 [docs/vendors/](../vendors/README.md)에 산다.

## 결과

- 벤더/계정 추가 = 설정 블록 추가. 코드 변경 0 (S3 호환인 한).
- 로컬(MinIO)과 프로덕션(OCI)이 같은 adapter, 같은 코드 경로.
- 잘못된 설정(깨진 참조, 자격증명 누락)은 부팅 실패. 런타임에 발견하지 않는다.

# ADR 001: provider는 코드가 아니라 설정이다

- Status: Accepted
- Date: 2026-07-02
- 부모: [000-identity](000-identity.md)

## 문제

filegate는 여러 벤더의 object storage 위에서 동작해야 한다. 벤더마다 접근 방식과 기능(사전 서명 접근, 대용량 분할 업로드, 아카이브 복원)이 다르고, 벤더 자체도 변한다 — MinIO는 2026년 초 OSS 저장소를 아카이브했다. 특정 벤더를 코드에 박으면 그 벤더의 운명이 filegate의 운명이 된다.

동시에 000의 원칙에 따라 서비스는 provider를 몰라야 한다. 그렇다면 "어떤 파일이 어떤 provider로 가는가"라는 지식은 어딘가에 살아야 하는데, 코드에 살면 배포 없이 못 바꾸고, 서비스에 살면 000 위반이다.

## 결정

**provider는 filegate 설정의 항목이다. 코드는 벤더를 모르고, adapter 계약만 안다.**

설정은 provider의 목록, 접근 정보, 그리고 각 provider가 무엇을 할 수 있는지(capability)를 선언한다. 코드는 그 선언을 읽어 동작할 뿐, 벤더별 분기를 갖지 않는다.

**S3 호환 계약을 1차 추상화로 삼는다.** object storage 세계의 사실상 표준 와이어는 S3 호환 API이고, 주요 벤더가 모두 이를 제공한다. llmgate가 OpenAI 와이어를 1차 계약으로 삼은 것과 같은 판단이다 — 표준 와이어 하나로 벤더 대부분을 커버하고, 벤더 네이티브 계약은 필요가 증명될 때만 추가한다.

**내부 접근 주소와 공개 접근 주소를 구분하는 개념을 처음부터 둔다.** filegate가 스토리지를 보는 경로(내부망)와 클라이언트가 스토리지에 접근하는 경로(공개망)는 다를 수 있고, 사전 서명된 접근은 주소에 묶인다. 이 구분은 특정 환경의 꼼수가 아니라 아키텍처 개념이다.

**capability는 선언식이다.** filegate가 벤더 기능을 런타임에 탐지하지 않는다. 운영자가 provider별로 선언하고, 배치와 접근 허가는 선언된 범위 안에서만 동작한다. 선언이 틀리면 해당 provider에서 실패가 드러나는 것이지, filegate가 추측하지 않는다.

**파일의 정체성과 물리 위치를 분리해 모델링한다.** 파일이 무엇인가(소유 client, 의도, 상태)와 파일이 어디에 있는가(provider, 위치)는 별개의 것이며, 위치는 파일의 생애 동안 바뀔 수 있는 값이다. 현재 위치는 포인터일 뿐이다. 이 분리가 있어야 provider 간 이동, 저비용 계층으로의 강등 같은 미래가 스키마 재설계 없이 열린다.

**배치(placement)는 정책의 소관이며, 처음에는 정적 매핑으로 시작한다.** 의도→provider의 단순 매핑에서 출발하되, 배치 결정은 인터페이스 뒤에 숨긴다. 비용·접근 패턴 기반의 동적 배치는 그 인터페이스의 내부 교체로 진화한다.

### 경계선

- **자동 비용 최적화와 자동 이동은 초기 범위가 아니다.** 그것은 백그라운드 조정자(reconciler)의 관할이며, 요청 경로에는 어떤 단계에서도 넣지 않는다.
- **벤더 네이티브 adapter는 필요가 증명될 때 추가한다.** S3 호환 계약으로 안 되는 기능(예: 아카이브 복원)이 실제로 필요해지기 전까지 만들지 않는다.
- **관리 공간 독점 규칙(000)이 여기에도 적용된다.** provider를 등록한다는 것은 그 저장 공간의 이름 체계를 filegate가 독점한다는 뜻이다. 객체의 이름은 클라이언트가 준 이름이 아니라 filegate가 발급한 불투명한 식별자다.

## 근거

- **llmgate가 같은 패턴을 검증했다.** 모델 선택을 설정으로 옮기자 벤더 추가가 설정 변경으로 끝났다. storage provider도 같은 성질의 문제다.
- **벤더 변동은 실재하는 리스크다.** MinIO 아카이브가 이 설계를 논의하는 도중에 실제로 일어났다. 이 설계에서 그 사건의 비용은 설정 항목 하나의 교체다. 이것이 이 ADR이 사는 이유다.
- **선언식이 탐지식보다 예측 가능하다.** 벤더 기능을 탐지하는 코드는 벤더 수만큼 늘어나는 유지비용이고, 운영자가 이미 아는 사실을 코드가 재발견하는 낭비다. 잘못된 선언은 부팅 시점이나 해당 provider 사용 시점에 드러난다.

## 결과

- 새 벤더 추가 = 설정에 provider 선언 추가. 코드 변경 0 (S3 호환인 한).
- 로컬 개발과 프로덕션이 같은 adapter, 같은 코드 경로를 탄다. 개발 환경용 특수 분기가 없다.
- 잘못된 설정(존재하지 않는 provider 참조, 접근 정보 누락)은 부팅 실패다. 런타임에 발견하지 않는다.
- 벤더 고유 기능은 capability로 선언되기 전까지 존재하지 않는 것으로 취급된다.

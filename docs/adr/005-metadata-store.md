# ADR 005: 메타데이터 저장소는 PostgreSQL이다

- Status: Draft
- Date: 2026-07-08
- 부모: [000](000-identity.md) 공리 3 (자기 메타데이터 저장소), [002](002-lease-model.md) lease 원장, [004](004-config-layers.md) 회계

## 문제

filegate의 진실은 전부 메타데이터다: file/location, lease 원장, capacity 회계. 쓰기 발급은 여러 후보 provider에 걸친 capacity 예약과 file_id 발급을 한 번에 해야 하고, 실패하면 전부 없던 일이 되어야 한다. 상태 전이는 동시 요청과 경합한다. 이 진실을 어디에 둘지 정해야 한다.

## 결정

- **메타데이터 저장소는 PostgreSQL 하나다.** 공리 3의 "자기 메타데이터 저장소"가 이것이다. filegate 전유이며 다른 서비스와 공유하지 않는다.
- **회계의 원자성은 트랜잭션이 보증한다.** 예약·정산·해제는 단일 트랜잭션 경계 안에서 한다. 애플리케이션 로직으로 원자성을 흉내 내지 않는다. quota 축이 도입되면 같은 트랜잭션 경계가 두 축을 함께 덮는다.
- **상태 전이는 조건부 갱신으로 한다.** pending → active → deleted는 현재 상태를 조건으로 거는 갱신이다. 동시 commit은 한쪽만 이긴다.
- **lease 원장이 곧 감사 기록이다.** 별도 감사 테이블을 만들지 않는다. 중계 토큰 검증도 이 원장 조회다.
- **reconciler도 같은 PG를 본다.** 별도 큐 시스템을 두지 않는다 (공리 3).

## 경계선

- 바이트는 PG에 넣지 않는다. 파일 내용의 자리는 provider뿐이다.
- 캐시 계층은 병목이 확인될 때만 추가한다.
- 스키마는 구현의 영역이다. 이 ADR은 저장소 선택과 원자성 원칙만 고정한다.

## 결과

- 배포 = 바이너리 + PostgreSQL + 설정 파일.
- PG 장애 시 새 발급은 실패한다. 이미 발급된 직결 접근은 계속된다 (000의 장애 성질과 대칭).
- 중계 모드는 바이트 요청마다 원장 조회가 경로에 들어간다. 중계는 설계상 예외(직결이 계약 기준)이지 저트래픽이라는 뜻은 아니다 — v0의 로컬 파일시스템처럼 중계 전용 provider가 트래픽을 지면 조회 비용이 실재하며, 병목이 확인되면 캐시로 다룬다.

# ADR 004: intent는 서비스의 어휘고, 배치는 운영자의 카탈로그다

- Status: Draft
- Date: 2026-07-03 (개정 2026-07-09: 등록부의 정본을 config에서 DB로)
- 부모: [000](000-identity.md) 공유 전제(여러 서비스가 하나의 filegate를 쓴다), 공리 1 (소유 분리), 공리 3 (내부 의존 0)

## 문제

filegate는 여러 서비스가 쓴다 (000의 전제). intent를 전역으로 두면 이름 충돌과 소유권 문제가 생긴다. 등록(누가 쓰나, 어디에 저장하나)의 정본이 필요하다. 운영자는 인프라 전체를 Terraform으로 선언 관리하고 있다.

## 결정

- **소유자가 다른 세 계층이다.** providers(물리 접근 계약) / profiles(운영자 배치 카탈로그) / clients(서비스 어휘: 인증 키 + 자기 intents).
- **등록부의 정본은 DB다.** 변경은 운영자 API를 통해서만 한다 (AWS 모델). 재배포 없이 등록이 바뀐다.
- **등록부에 비밀은 없다.** provider 자격증명은 규약 env(`FILEGATE_PROVIDER_<ID>_*`)로 ESO가 공급하고, 클라이언트 키는 sha256 해시로만 저장한다.
- **intent는 클라이언트 네임스페이스 안에 살고 profile을 가리킨다.** provider를 직접 가리키지 않는다 — 배치를 바꿔도 서비스 계약은 유지된다.
- **참조 무결성은 DB가 쓰기 시점에 집행한다.** 사용 중인 등록(참조되는 provider·profile, 파일 가진 client)의 삭제는 거부된다. 부팅 시 그래프 검증을 이것이 대체한다.
- **provider 등록은 그 자체가 검증이다.** 등록 시 env 자격증명 존재와 저장 공간 접근을 확인하고, 실패하면 등록을 거부한다.
- **capacity 상한은 provider 등록의 일부다.** 기본값은 무료 구간([vendors/](../vendors/README.md) 기준)이다. quota 집행은 다음 범위다 (회계 시점의 정본은 [spec 00](../spec/00-operations.md)).

## 경계선

- filegate는 intent 이름의 의미를 해석하지 않는다.
- 관리 화면은 두지 않는다. 표면은 운영자 API와 그 클라이언트다: CLI(사람), MCP(AI), Terraform provider(선언 관리).
- 운영자 API 인증은 정적 운영자 토큰(env) 하나다.
- 서버(프로세스) 설정은 전부 env다. YAML 설정 파일은 두지 않는다.
- 운영자 API가 생기기 전까지 v0의 등록 쓰기는 SQL(psql/opsgate)로 한다 — 정본이 DB라는 사실은 같다.

## 결과

- 새 서비스 온보딩 = 등록 호출 하나 (CLI `client create` 또는 `terraform apply`). 재배포 없음.
- 등록 이력·백업은 감사 기록과 export로 다룬다 (다음 범위). git은 등록부가 아니다.
- 클라이언트는 다른 클라이언트의 영역을 수정할 수 없다.
- 사용량 조회는 운영자 시점뿐이다. 클라이언트는 용량 정보를 받지 않는다 — 자기 사용량 관리가 필요한 서비스는 스스로 한다 (공리 1).

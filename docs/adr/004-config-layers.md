# ADR 004: 어휘는 서비스, 연결은 운영자, 정본은 DB다

- Status: Accepted
- Date: 2026-07-03 (개정 2026-07-09: 등록부의 정본을 config에서 DB로. 2026-07-10: 용어 provider→storage, profile 제거·binding 도입. 2026-07-13: capacity를 관찰 기준선으로)
- 부모: [000](000-identity.md) 공유 전제(여러 서비스가 하나의 filegate를 쓴다), 공리 1 (소유 분리), 공리 3 (내부 의존 0)

## 문제

filegate는 여러 서비스가 쓴다 (000의 전제). intent를 전역으로 두면 이름 충돌과 소유권 문제가 생긴다. 등록(누가 쓰나, 어디에 저장하나)의 정본이 필요하다. 운영자는 인프라 전체를 Terraform으로 선언 관리하고 있다.

## 결정

- **독립 노드 둘과 그것을 잇는 엣지 하나다.** storages(물리 접근 계약)와 clients(서비스 신원: 인증 키)는 서로를 모른다. bindings(client의 intent 이름 → storage)만이 둘을 참조한다.
- **등록부의 정본은 DB다.** 변경은 운영자 API를 통해서만 한다 (AWS 모델). 재배포 없이 등록이 바뀐다.
- **비밀의 저장 방식은 성격이 정한다.** 클라이언트 키는 검증만 필요하므로 sha256 해시로만 저장한다. storage 시크릿은 filegate가 서명에 원문을 써야 하므로 암호화(AES-GCM, 마스터 키는 env)해 등록부에 보관한다 — opsgate의 credential 보관 방식.
- **서비스 계약은 intent 이름이다.** binding의 storage 포인터는 운영자가 언제든 바꾼다 — 배치를 바꿔도 서비스 계약은 유지된다. 일괄 배치 변경은 Terraform 참조가 담당한다 (여러 binding이 같은 storage 참조를 공유).
- **참조 무결성은 DB가 쓰기 시점에 집행한다.** binding이 남아 있는 storage·client, 파일을 가진 client의 삭제는 거부된다 — 연결을 먼저 지워야 노드를 지울 수 있다. 부팅 시 그래프 검증을 이것이 대체한다.
- **storage 등록은 그 자체가 검증이다.** 등록 요청에 담긴 자격증명으로 저장 공간 접근을 즉석 확인하고, 실패하면 등록을 거부한다.
- **capacity는 storage 등록의 일부다 — 집행이 아니라 관찰 기준선이다** ([spec 00](../spec/00-operations.md)). 시스템 기본값은 없다 — 등록자가 정하며, 무료 구간([vendors/](../vendors/README.md))이 권장 시작값이다. quota 집행은 비목표다 (관찰이 대신한다).

## 경계선

- filegate는 intent 이름의 의미를 해석하지 않는다.
- 관리 화면은 두지 않는다. 표면은 운영자 API와 그 클라이언트다: Terraform provider(선언 관리), 필요 시 CLI.
- 운영자 API 인증은 정적 운영자 토큰(env, 쉼표 목록)이다 — 메인/서브 두 개로 무중단 로테이션한다.
- 서버(프로세스) 설정은 전부 env다. YAML 설정 파일은 두지 않는다. env의 비밀은 마스터 키·운영자 토큰·DB URL뿐이다.
- storage 등록은 처음부터 운영자 API로 한다 (시크릿 암호화 때문에 SQL로 못 만든다). clients·bindings는 API가 생기기 전까지 SQL로도 가능하다.

## 결과

- 새 서비스 온보딩 = 등록 호출 하나 (`terraform apply` 또는 이후 CLI). 재배포 없음.
- 등록 이력·백업은 감사 기록과 export로 다룬다 (다음 범위). git은 등록부가 아니다.
- 클라이언트는 다른 클라이언트의 영역을 수정할 수 없다.
- 사용량 조회는 운영자 시점뿐이다. 클라이언트는 용량 정보를 받지 않는다 — 자기 사용량 관리가 필요한 서비스는 스스로 한다 (공리 1).

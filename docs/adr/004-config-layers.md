# ADR 004: intent는 서비스의 어휘고, 배치는 운영자의 카탈로그다

- Status: Draft (최초 작성)
- Date: 2026-07-03
- 부모: [000-identity](000-identity.md) 공리 1 (소유 분리) + 공리 3 (최상위 — filegate는 여러 서비스가 쓴다)

## 문제

filegate는 한 서비스의 부속이 아니라 여러 서비스가 쓰는 게이트다. 그런데 초기 설정은 intent를 전역 풀로 두고 client가 이름으로 참조하는 구조였다 — `note_attachment`라는 notegate의 어휘가 전역 이름공간에 산다. 서비스가 늘수록 이름 충돌, 소유권 불명, 한 서비스의 변경이 남의 영역을 스치는 문제가 커진다.

용량에도 같은 문제가 있다. "용량"이라는 말에는 사실 두 개의 다른 진실이 섞여 있다: filegate가 각 client에게 **약속한 몫**(quota)과, 저장 공간 자체의 **물리적 한도**(capacity — 무료 구간, 디스크 크기). client 몫의 합이 물리 한도와 무관하게 설정될 수 있고, 물리 한도는 client가 몇이든 합산으로 차오른다. 한 축의 회계로는 둘 중 하나를 거짓말하게 된다.

## 결정

**설정을 세 계층으로 나눈다. 각 계층은 소유자가 다르다.**

```text
providers          물리 계층: 접근 계약의 목록          (인프라의 사실)
storage_profiles   정책 계층: 배치 방식의 카탈로그       (운영자의 결정)
clients            소비자 계층: 서비스별 자기완결 블록    (각 서비스의 어휘)
```

**intent는 client 블록 안에 산다.** intent는 서비스가 자기 파일의 용도를 말하는 어휘이므로, 이름공간도 서비스별이다. 서로 다른 client가 같은 intent 이름을 써도 무관하다. client 블록은 자격증명, 몫(quota), 그리고 자기 intent들을 담은 자기완결 단위이며, 서비스 추가는 블록 하나의 추가다.

**intent는 배치를 직접 말하지 않고 profile을 가리킨다.** 어느 저장 공간에, 어떤 수명(lease TTL, 보존 기간)으로 둘지는 운영자가 storage profile로 정의하고, intent는 그중 하나를 참조한다. 서비스의 어휘와 운영자의 배치 결정이 분리되므로, 배치를 바꿔도 서비스 쪽 계약(intent 이름)은 흔들리지 않는다.

**회계는 두 축이다.** 쓰기 lease 발급 시 client quota(약속)와 provider capacity(물리) **양쪽에** 예약을 걸고, 어느 한쪽이라도 초과면 발급을 거부한다. 확정 시 양쪽 모두 실측으로 정산하고, 만료·취소 시 양쪽 모두 해제한다. 하나의 물리적 사건(바이트가 놓인다)이 두 장부에 기록되는 것이며, 두 장부는 답하는 질문이 다르다 — quota는 "이 서비스가 약속을 지키고 있나", capacity는 "이 공간이 차오르고 있나".

**capacity의 기본값은 무료 구간이다.** 물리 한도의 출처(무료 구간, 디스크 크기, 예산)는 변하는 외부 사실이므로 벤더 노트(`docs/vendors/`)가 담고, 설정은 그 시점의 값을 박는다. capacity 압박은 훗날 배치 이동(tiering)의 트리거가 된다.

### 경계선

- **filegate는 intent 이름의 의미를 해석하지 않는다.** `note_attachment`가 무엇인지 아는 것은 notegate뿐이다. filegate에게 intent는 profile로 가는 포인터이자 회계의 키일 뿐이다.
- **오버커밋은 금지가 아니라 가시화 대상이다.** client quota의 합이 capacity를 넘는 설정은 유효하다(모두가 몫을 다 쓰지는 않는다). 다만 부팅 시 그 사실을 드러낸다. 물리가 먼저 차면 quota가 남아 있어도 발급은 거부된다 — 물리는 거짓말하지 않는다.
- **잘못된 참조는 부팅 실패다.** 존재하지 않는 profile을 가리키는 intent, 존재하지 않는 provider를 가리키는 profile은 실행 중에 발견하지 않는다.
- **관리 화면은 없다.** 세 계층 모두 배포 설정이 진실이다(000). 계층화는 UI의 대체물이 아니라 UI가 필요 없게 만드는 수단이다.

## 근거

- **소유자가 다른 것을 한 이름공간에 두면 소유권이 사라진다.** intent의 주인은 서비스, profile의 주인은 운영자, provider의 주인은 물리 세계다. 계층 분리는 각 변경이 누구의 결정인지를 구조로 말한다.
- **약속과 물리는 실제로 다른 축이다.** 무료 구간(물리)은 client 수와 무관하게 하나이고, client 몫(약속)은 물리와 무관하게 협상된다. 두 축이 겹치는 유일한 지점이 lease 발급이므로, 그곳에서 양쪽을 본다.
- **서비스 어휘와 배치의 분리는 이동의 전제다.** intent가 provider를 직접 가리키면 배치 변경이 서비스 계약 변경이 된다. profile 간접층이 있어야 "notegate는 모르는 채로 note_attachment를 다른 곳에 두기"가 성립한다 — 공리 1의 설정판.

## 결과

- 새 서비스 온보딩 = clients에 블록 하나. 다른 client의 영역은 물리적으로 건드릴 수 없다.
- intent 이름 충돌이라는 개념 자체가 사라진다.
- 사용량 조회는 두 시점을 갖는다: client는 자기 몫의 소진을, 운영자는 공간별 합산을 본다.
- capacity 축은 훗날 "차오르면 안 읽히는 것부터 밀어낸다"(tiering)의 판단 기반이 된다. 그 판단의 근거 데이터(파일별 접근 빈도)는 이미 lease 기록에 있다(002).
- 초기 구현(전역 intent 풀, 단일 축 usage_counters)은 이 구조로 재편된다.

# ADR 002: 모든 바이트 접근은 lease다

- Status: Accepted
- Date: 2026-07-02 (개정 2026-07-03: capability 보안 모델 명시)
- 부모: [000-identity](000-identity.md) 공리 2 (바이트 불통과 — filegate는 컨트롤 플레인이다)

## 문제

000에 따라 바이트는 filegate를 통과하지 않는다. 그러면 클라이언트가 스토리지와 직접 주고받는 접근을 filegate가 어떻게 허가하고, 기록하고, 사후 검증하는가가 문제가 된다.

먼저 기각한 대안: **스토리지 API 흉내(facade)** — 서비스가 표준 스토리지 API로 filegate를 직접 바라보는 방식. 표준 스토리지 프로토콜에는 "다른 곳에 가서 전송하라"는 우회가 없어 모든 바이트가 filegate를 통과해야 하고, 저장 위치의 이름을 서비스가 정하게 되어 000이 깨지며, 의도를 표현할 자리도 없다. 이 길을 간 프로젝트들은 유지에 실패했다.

남는 모델은 사전 서명된 직접 접근(presigned)인데, 이걸 그대로 쓰면 세 가지 문제가 있다.

1. **허가는 미래의 행위에 대한 것이고, 스토리지는 약속을 강제하지 않는다.** 접근을 허가한 순간 선언과 다른 것(더 큰 파일, 다른 내용)이 올라와도 막을 수 없다.
2. **전송이 끝났는지 filegate가 모른다.** 허가만 하고 잊으면 기록과 실제 객체가 어긋난다 — 고아 객체, 유령 메타데이터.
3. **업로드용 개념과 다운로드용 개념을 따로 만들면** 수명주기가 둘로 늘고, 감사·정리·접근 추적이 각각 따로 논다.

## 결정

**바이트 플레인에 대한 모든 접근을 하나의 개념으로 통일한다.**

> lease = 특정 파일에 대한, 시간제한이 있고, 취소 가능한, 단일 목적의 접근 권한

쓰기든 읽기든, 서비스가 쓰든 서비스가 최종 사용자에게 위임하든, 스토리지에 닿는 모든 접근은 lease 하나를 통해 일어난다.

**쓰기는 2단계다: 허가, 그리고 확정.** 서비스가 의도와 함께 파일을 선언하면 filegate가 배치를 결정하고 쓰기 lease를 발급한다. 전송은 클라이언트와 스토리지가 직접 한다. 전송 후 서비스가 확정(commit)을 요청하면, filegate가 실제 객체를 스토리지에서 확인해 선언과 대조한다. 검증을 통과해야 파일이 존재하는 것으로 인정된다.

**확정이 검증 게이트다.** 허가 시점에 강제할 수 없는 약속(크기, 완결성)을 확정 시점에 걸러낸다. 확정되지 못한 바이트는 파일이 되지 못하고 회수 대상이 된다. 용량 관리도 같은 지점에 얹힌다 — 허가 시 선언량을 예약하고, 확정 시 실측으로 정산하고, 만료·취소 시 예약을 해제한다.

**읽기는 현재 위치를 숨기는 간접층이다.** 서비스가 파일 식별자로 읽기 lease를 요청하면 filegate가 그 시점의 실제 위치를 찾아 접근을 발급한다. 위치가 바뀌어도 서비스는 모른다.

**lease는 위임 가능하다.** 서비스가 lease를 최종 사용자에게 넘기면 사용자가 스토리지와 직접 통신한다. filegate도 서비스도 바이트를 만지지 않는다 — 게이트라는 이름의 의미가 여기 있다.

**lease가 유일한 감사 단위다.** 바이트 플레인의 모든 접근이 lease 하나에 대응한다. 접근 추적, 고아 탐지("만료된 쓰기 lease + 확정 안 됨"), 감사 기록이 전부 한 개념에서 나온다. lease의 생애(발급 → 확정 | 만료 | 취소)가 시스템의 심장이다.

**보안 모델: 검문은 발급 앞에, 열쇠는 짧고 좁게.** lease가 낳는 접근 수단은 인증이 아니라 소지(capability)다 — 가진 자가 곧 통과한다. 그 안전은 세 겹에서 나온다: 추측 불가능(암호학적 서명), 짧은 수명(만료 후 저장소가 거부), 좁은 범위(객체 하나, 동작 하나). 파일이 일시적으로 공개되는 것이 아니라, 그 파일 하나로 통하는 문이 잠시 생겼다 사라지는 것이다. 진짜 보안 경계는 접근 수단이 아니라 그것이 발급되기 *전* — 서비스의 사용자 권한 검사와 filegate의 client 검사 — 에 있으며, 접근 수단 자체는 이미 내려진 허가의 배달 통로일 뿐이다.

### 경계선

- **filegate는 허가의 실사용 시점을 모른다.** 스토리지는 알려주지 않는다. lease의 진실은 "발급했다"와 "확정됐다/만료됐다"뿐이며, 그 사이를 아는 척하지 않는다.
- **취소(revoke)는 이미 나간 허가를 되부르지 못한다.** 스토리지의 서명은 유효기간까지 산다. 취소의 실효는 "확정을 거부한다"와 "만료 후 회수한다"이다. 따라서 lease 유효기간은 짧게 잡는 것이 기본 철학이다.
- **읽기는 용량을 소비하지 않는다.** 용량은 물리적 점유를 추적한다. 삭제가 결정됐어도 실제로 회수되기 전까지는 점유다.
- **재개 가능한 업로드, lease 갱신은 초기 범위가 아니다.** 필요가 증명될 때 lease 개념의 확장으로 추가한다.

## 근거

- **업계에서 검증된 패턴이다.** Slack과 Notion의 파일 API가 같은 2단계 구조(허가 → 직접 전송 → 확정)를 쓰고, tus 프로토콜이 "전송 세션 = 상태를 가진 리소스" 모델을 검증했다. 우리는 그 둘을 lease 하나로 접었다.
- **개념 하나가 셋을 산다.** 업로드 세션, 다운로드 허가, 감사 레코드를 따로 만들면 세 개의 수명주기를 관리해야 한다. 통일하면 정리·감사·접근 추적이 한 개념의 조회가 된다.
- **가용성 특성이 좋다.** filegate 장애 시 새 허가만 멈춘다. 이미 나간 허가는 스토리지가 직접 검증하므로 진행 중인 전송은 계속된다. 바이트 플레인은 애초에 filegate에 의존하지 않는다.

## 결과

- API 표면이 작게 유지된다: 허가(쓰기/읽기), 확정, 조회, 삭제 결정 — 이것이 전부다.
- 요청 경로는 "정책 확인 → 배치 결정 → lease 발급 → 기록"으로 단순하게 유지되고, 무거운 일(회수, 검증, 보존 집행)은 전부 백그라운드 조정자로 간다.
- 약속을 어긴 전송은 존재할 수는 있으나 파일이 될 수 없고, 점유한 바이트는 만료 후 회수된다.
- 서비스는 lease로 받은 접근 수단을 불투명한 것으로 취급한다. 그 내부 구조에 의존하는 서비스는 계약 위반이다.

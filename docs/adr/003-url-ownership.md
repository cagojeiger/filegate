# ADR 003: 안정 URL은 서비스의 것이고, filegate의 URL은 소모품이다

- Status: Accepted
- Date: 2026-07-03
- 부모: [000-identity](000-identity.md), [002-lease-model](002-lease-model.md)

## 문제

002에 따라 filegate가 발급하는 접근 수단(presigned URL)은 짧게 살고 계속 바뀐다. 그런데 최종 사용자의 세계는 붙박이 URL을 요구한다 — 문서에 박아두는 이미지 주소, 공유되는 링크, 북마크. 이 안정성을 누가 제공하는가?

filegate가 공개 다운로드 주소(`/files/{id}/download` 같은 것)를 직접 열면 간단해 보인다. 하지만 그 주소에는 "이 사용자가 이 파일을 봐도 되는가"를 판단할 방법이 없다 — filegate는 사용자를 모른다(000). 즉 filegate의 공개 URL은 서비스의 권한 검사를 우회하는 뒷문이 된다. 편의를 위해 정체성을 깨는 길이다.

## 결정

**안정 URL은 서비스 계층이 소유한다. filegate는 공개 표면을 갖지 않는다.**

최종 사용자를 마주하는 URL은 서비스의 도메인에 산다. 서비스는 자기 방식으로 사용자를 인증하고, 자기 도메인 지식으로 권한을 확인한 뒤, filegate에서 읽기 lease를 받아 사용자를 실제 저장소로 넘긴다(redirect). 사용자에게 URL은 영원히 같고, 그 뒤에서 접근 수단은 매번 새로 만들어진다.

```text
사용자 → 서비스의 안정 URL     (사용자 인증, 권한 확인 — 서비스 몫)
       → filegate read lease   (위치 해석, 접근 발급, 기록 — filegate 몫)
       → 저장소로 redirect      (바이트는 여전히 직접 흐른다)
```

**filegate의 모든 API는 등록된 client 인증 뒤에 있다.** 익명의 브라우저가 filegate에 직접 닿는 경로는 존재하지 않는다.

**filegate가 발급한 URL은 소모품이다.** 서비스는 그것을 저장하지 않고, 문서에 박지 않고, 캐시에 태우지 않는다. 서비스가 영속화하는 유일한 것은 파일 식별자다. 죽어가는 URL이 어딘가에 박제되는 순간 이 모델은 깨진다.

**접근은 매번 그 시점의 진실로 해석된다.** 읽기 lease를 발급할 때마다 filegate는 파일의 현재 위치를 새로 조회한다. 이 재해석이 있어야 위치 이동(migration)이 사용자 모르게 일어날 수 있다 — 안정 URL과 움직이는 바이트가 양립하는 유일한 방법이다.

**표현은 발급 시점에 서비스가 명시한다.** 사용자에게 보일 파일명, 표시 방식(inline/attachment)은 읽기 lease를 요청할 때 서비스가 지정하고, filegate는 그것을 접근 수단에 새겨 넣는다. filegate가 보관 중인 클라이언트 메타데이터를 해석해서 추측하지 않는다(000의 불투명 원칙 유지).

### 경계선

- **redirect 홉은 서비스의 비용이다.** 사용자의 모든 접근이 서비스를 한 번 경유한다. 이것은 버그가 아니라 권한 검사가 사는 자리다. 이 홉이 병목이라는 증거가 생기면, 서비스가 권한 확인 후 filegate제 단기 위임 토큰을 받아 사용자에게 주는 확장이 가능하다 — 권한 판단이 발급 시점에 서비스에 남는 한 이 ADR과 충돌하지 않는다. 증거 없이 미리 만들지 않는다.
- **캐시는 안정 URL까지만.** 서비스의 안정 URL 응답(redirect)은 접근 수단의 수명보다 오래 캐시되어서는 안 된다. CDN을 태울 대상은 안정 URL이 아니라, 필요하다면 그 뒤의 저장소 계층이다.
- **filegate는 익명 트래픽의 인증 게이트가 되지 않는다.** "filegate가 유저 토큰을 검증해주면 되지 않나"는 유혹은 000 위반이다. 사용자 개념은 어떤 형태로도 filegate에 들어오지 않는다.

## 근거

- **000의 직접 귀결이다.** 권한 검사가 영원히 서비스 몫이라면, 사용자를 마주하는 표면도 서비스 몫이다. 표면과 권한 검사는 같은 곳에 살아야 한다.
- **002의 짧은 수명 철학과 안정성 요구의 화해다.** 유출된 접근 수단의 피해 반경을 lease 수명으로 묶어두면서도, 사용자에게는 영구적인 주소를 줄 수 있다 — 안정성을 소모품 위의 간접층으로 만들면 된다.
- **업계에서 검증된 형태다.** GitHub release asset, Slack 파일 링크가 같은 구조다: 안정 주소는 서비스가 들고, 실제 전송은 만료되는 서명 URL로 그때그때 넘긴다.
- **감사가 공짜로 따라온다.** 모든 사용자 접근이 lease 발급 한 건으로 남으므로, 접근 추적(레벨링, 비용 판단의 근거)이 별도 장치 없이 쌓인다.

## 결과

- 각 서비스는 자기 도메인에 파일 접근 라우트를 갖는다. filegate 도입이 서비스의 URL 체계를 바꾸지 않는다.
- filegate에는 사용자 개념이 영원히 없다. 인증 모델은 client 하나로 유지된다.
- URL 유출의 피해 반경은 lease 수명이다. 영구 유출이라는 개념 자체가 없다.
- 서비스 저장소에 filegate가 발급한 URL이 영속화되어 있다면 그것은 계약 위반이다. 검증 포인트: 서비스 DB에는 파일 식별자 외의 filegate 산출물이 없어야 한다.
- 위치 이동은 언제나 사용자 모르게 가능하다. 안정 URL이 그 은폐를 완성한다.

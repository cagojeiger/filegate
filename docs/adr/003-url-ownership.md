# ADR 003: 안정 URL은 서비스의 것, filegate URL은 소모품

- Status: Draft (최초 작성)
- Date: 2026-07-03
- 부모: [000](000-identity.md) 공리 1+2의 교차, [002](002-lease-model.md)

## 문제

접근 수단은 짧게 살고 계속 바뀌는데(002), 사용자의 세계는 붙박이 URL을 요구한다 — 문서에 박는 이미지, 공유 링크. filegate가 공개 다운로드 주소를 열면 간단해 보이지만, 그 주소엔 유저 권한을 판단할 방법이 없어(공리 1) 서비스의 권한 검사를 우회하는 뒷문이 된다.

## 결정

- **사용자를 마주하는 URL은 서비스 도메인에 산다.** 서비스가 유저 인증 → 권한 확인 → filegate에서 읽기 lease 수령 → 저장소로 redirect. 사용자에게 URL은 영원히 같고, 뒤에서 접근 수단은 매번 새로 만들어진다.
- **filegate는 공개 표면이 없다.** 모든 API는 등록된 client 인증 뒤. 익명 브라우저가 filegate에 직접 닿는 경로는 존재하지 않는다.
- **filegate가 발급한 URL은 소모품.** 저장·임베드·캐시 금지. 서비스가 영속화하는 것은 file_id뿐.
- **표현은 발급 시점에 명시.** 사용자에게 보일 파일명·표시 방식은 lease 요청 때 서비스가 지정한다. 보관된 메타데이터를 filegate가 해석해 추측하지 않는다.

## 경계선

- **redirect 홉은 비용이 아니라 권한 검사가 사는 자리다.** 병목이 증명되면 위임 토큰(서비스가 권한 확인 후 발급받아 사용자에게 전달) 확장이 가능 — 권한 판단이 서비스에 남는 한 이 ADR과 충돌하지 않는다.
- **"공개 파일" 개념은 filegate에 없다.** 공개란 서비스의 권한 검사가 "누구나 통과"인 것이지 저장소가 공개되는 게 아니다. (Notion도 공개 페이지에 만료되는 서명 URL을 그때그때 발급한다.)
- **유저 토큰 검증 대행 금지.** "filegate가 유저 토큰을 봐주면 되잖나"는 000 위반.

## 결과

- filegate 도입이 서비스의 URL 체계를 바꾸지 않는다. 유저 개념도 영원히 안 들어온다.
- URL 유출의 피해 반경 = lease 수명. 영구 유출이라는 개념이 없다.
- 위치 이동은 언제나 사용자 모르게 가능 — 안정 URL이 은폐를 완성한다.
- 검증 포인트: 서비스 DB에 file_id 외의 filegate 산출물(URL 등)이 있으면 계약 위반.

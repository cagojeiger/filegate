# ADR 000: 서비스는 파일의 물리적 존재를 관리하지 않는다

- Status: Accepted
- Date: 2026-07-02
- 관련: 001 ~ 003 (이 원칙에서 파생)

## 문제

서비스가 파일을 다루면 task와 무관한 인프라가 따라 들어온다 — 스토리지 벤더 선택, 자격증명 관리, 업로드/다운로드 경로, 고아 객체 정리, 보존 기간 집행. 서비스마다 같은 인프라를 다시 짜고 또 짠다.

또 *어디에 저장하는지*가 서비스에 박힌다. 파일을 다루려고 서비스가 구체 위치(벤더, 버킷, 키)를 들고 있어야 하고, 스토리지를 옮기려면 서비스 N개를 동시에 고쳐야 한다. 벤더 자격증명이 서비스 수만큼 흩어져 유출 표면이 넓어지고, 스토리지 비용이 어디서 새는지 아무도 전체를 못 본다.

생태계에는 이미 같은 문제를 푼 선례가 있다. authgate(앱은 인증 인프라를 관리하지 않는다), llmgate(호출자는 모델 호출 인프라를 관리하지 않는다), opsgate(LLM 클라이언트는 secret을 직접 보지 않는다) — 각각 한 종류의 횡단 관심사를 흡수한 인프라 게이트다.

## 결정

**filegate는 네 번째 인프라 게이트다. 서비스는 파일의 식별자와 의도만 다룬다.**

파일의 **의미**(누구의 파일인지, 무엇에 붙었는지, 어떤 사용자가 볼 수 있는지)는 서비스가 소유하고, 파일의 **물리적 존재**(어디에, 어떤 형태로, 언제까지, 얼마짜리로 있는지)는 filegate가 소유한다.

서비스는 파일을 만들 때 저장 위치가 아니라 **의도(intent)** — 이 파일이 어떤 용도인지 — 를 전달한다. 실제 저장 위치는 filegate가 정책으로 결정하고, 서비스에는 파일 식별자 하나만 남는다. 식별자는 물리 세계로 가는 외래키다.

**바이트는 filegate를 통과하지 않는다.** filegate는 접근을 허가하고, 기록하고, 사후에 책임지는 컨트롤 플레인이며, 실제 전송은 클라이언트와 스토리지가 직접 한다.

**인프라 계층 최상위를 유지한다.** filegate는 생태계 내부의 어떤 것에도 의존하지 않는다:

- authgate를 호출하지 않는다. 호출자 인증은 filegate가 자체 관리하는 정적 자격증명으로 한다. authgate는 사람용이고, filegate의 호출자는 서비스다.
- llmgate, opsgate도 모른다. 의존 대상은 스토리지 벤더와 자기 자신의 메타데이터 저장소뿐이다.
- notegate가 filegate를 아는 것이지 반대가 아니다. filegate에게 서비스는 등록된 client 이름 하나다.

llmgate와 달리 무상태 실행기가 아니다. 메타데이터 — 파일의 물리적 진실 — 가 제품의 본체이므로, filegate는 자기 저장소를 가진 상태형 서비스다.

### 경계선

오해를 피하기 위해 네 가지를 명시한다.

- **사용자 권한 검사는 영원히 서비스 몫이다.** "이 유저가 이 파일을 봐도 되나"를 filegate가 판단하려면 모든 서비스의 유저 모델을 알아야 하고, 그 순간 게이트가 아니라 플랫폼이 된다. filegate는 "이 client가 이 파일에 접근을 요청했다"까지만 판단한다.
- **클라이언트 메타데이터는 불투명하다.** 서비스가 맡긴 파일명이나 부가 정보는 저장하고 그대로 돌려줄 뿐 절대 해석하거나 쿼리하지 않는다. 쿼리하는 순간 filegate가 서비스들의 DB가 된다.
- **filegate가 관리하는 저장 공간은 100% filegate 소유다.** 정리(GC)는 "기록에 없는 객체 = 쓰레기"를 전제하므로, 관리 공간에 대한 외부 직접 접근은 데이터 파괴다. 공유는 없다 — 전부이거나 전무다.
- **모든 파일 저장이 filegate를 써야 하는 것은 아니다.** 관할은 object storage에 들어갈 파일이다. 서비스 자체 DB의 인라인 데이터, 배포 파이프라인의 정적 자산, filegate 자신의 백업(닭과 달걀)은 관할 밖이고, 채택은 서비스별 opt-in이다.

## 근거

- **서비스의 본분은 도메인이다.** 스토리지 인프라는 서비스마다 반복할 일이 아니다.
- **위치 독립성.** 서비스가 식별자만 알면 filegate가 뒤에서 파일을 옮겨도 아무 서비스도 변경되지 않는다. 위치가 서비스에 박히는 순간 이동은 사실상 불가능해진다.
- **자격증명 집중.** 벤더 자격증명은 filegate 한 곳에만 존재한다. 서비스는 filegate 접근 권한만 가진다.
- **결정과 집행의 분리.** "지워도 된다"는 서비스가 결정하고, "언제 어떻게 지울지"는 filegate가 집행한다. 보존 기간 안에는 복구 여지가 남는다.
- **이런 독립 오픈소스는 없고, 그게 정상이다.** 필요한 조직은 전부 사내 인프라로 만들었고 공개할 동기가 없다. 원래 각자 자기 것을 만드는 종류의 인프라이고, 우리 조건(호출자가 전부 우리 서비스, llmgate로 같은 패턴 검증)에 맞다.

## 결과

- 서비스에는 물리 위치가 존재하지 않는다. 파일 식별자만 남는다.
- 스토리지 벤더 추가/교체는 filegate 설정 변경으로 끝난다. 서비스 변경 0.
- filegate 장애는 "새 접근 허가 불가"이지 "파일 접근 불가"가 아니다. 이미 허가된 접근은 스토리지가 직접 검증하므로 계속 동작한다.
- 사용자 인증·권한, 파일의 도메인 의미는 filegate 밖에 머문다.

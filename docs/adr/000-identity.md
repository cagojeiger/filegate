# ADR 000: 서비스는 파일의 물리를 관리하지 않는다

- Status: Draft
- Date: 2026-07-03

## 문제

서비스마다 스토리지 인프라를 직접 만들면 중복이 커진다. 벤더 SDK, 자격증명, presigned URL, GC, 보존 집행이 서비스마다 흩어진다. 파일 위치가 서비스에 저장되면 이동도 어렵다. 그리고 여러 서비스가 제한된 저장 공간(무료 구간, 디스크)을 나눠 쓰려면 사용량 회계가 한 곳에 있어야 한다 — 이것이 filegate의 중심 책임이다.

## 결정: 세 공리

전제: 하나의 filegate 배포를 여러 서비스가 공유한다. 그 위에서 세 공리는 서로 독립된 선택이다. 파생 ADR은 이 공리를 부모로 둔다.

**공리 1. 소유 분리:** 파일의 의미는 서비스가 소유한다. 파일의 물리는 filegate가 소유한다. 서비스는 intent로 요청하고 file_id로 참조한다. 서비스가 저장하는 filegate 산출물은 file_id뿐이다.

**공리 2. 바이트 직결이 기본:** 바이트는 전송 주체와 저장소 사이에서 직접 오간다. filegate는 발급, 기록, 사후검증만 한다. 직접 접근이 불가능할 때만 filegate 중계를 허용한다. 중계는 capability가 강제할 때만 쓴다.

**공리 3. 내부 의존 0:** filegate는 저장소 벤더와 자기 메타데이터 저장소에만 의존한다. 호출자 인증은 자체 정적 키로 한다. 서비스가 filegate를 알고, filegate는 서비스를 모른다.

## 경계선

- 사용자 권한 검사는 서비스가 한다. filegate에는 유저 개념이 없다.
- 삭제는 결정·집행 분리다. detach 결정은 서비스가 내리고, 물리 purge는 filegate가 집행한다.
- 클라이언트 메타데이터는 불투명하다. 저장·반환만 하고 해석하지 않는다.
- 관리 공간은 전부이거나 전무다. 외부 직접 접근은 허용하지 않는다.
- 채택은 opt-in이다. 서비스 DB 인라인 데이터, CDN 자산, filegate 백업은 관할 밖이다.

## 결과

- 서비스에는 file_id만 남는다.
- 벤더 교체는 설정 변경으로 처리한다.
- filegate 장애 시 새 발급은 실패한다. 이미 발급된 접근과 진행 중 전송은 계속된다.

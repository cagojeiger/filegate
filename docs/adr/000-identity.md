# ADR 000: 서비스는 파일의 물리적 존재를 관리하지 않는다

- Status: Draft (최초 작성 — 실 운영 검증 후 Accepted 전환)
- Date: 2026-07-03

## 문제

서비스마다 스토리지 인프라(벤더 SDK, 자격증명, presigned, GC, 보존 집행)를 다시 짜고, 물리 위치가 서비스에 박혀 이동이 불가능해지며, 벤더 키가 서비스 수만큼 흩어진다.

## 결정 — 세 공리

서로 연역되지 않는 독립 선택이므로 셋 다 명시한다. 모든 파생 ADR은 이 공리들을 부모로 둔다.

**공리 1. 소유 분리** — 파일의 *의미*(누구의, 무엇에 붙은, 누가 보나, 언제 지우나)는 서비스가, *물리*(어디에, 언제까지, 얼마로)는 filegate가 소유한다. 경계를 오가는 것은 intent와 file_id뿐이며, file_id가 서비스가 영속화하는 유일한 filegate 산출물이다.

**공리 2. 바이트 직결이 기본** — 전송은 클라이언트↔저장소 직접. filegate는 허가·기록·사후검증만 하는 컨트롤 플레인이다. 저장소가 직접 접근을 제공할 수 없을 때(presigned 불가, 브라우저 CORS 불가)에 한해 filegate가 바이트를 중계하는 **선언된 예외**를 허용한다 — 예외는 capability가 강제할 때만이며, 편의로 켜지 않는다. (Slack은 전면 프록시를 선택하고도 성립한다. 즉 이것은 법칙이 아니라 우리 규모에 맞는 선택이고, 그래서 공리로 박는다.)

**공리 3. 내부 의존 0** — 의존은 저장소 벤더와 자기 메타데이터 저장소뿐. 호출자 인증은 자체 정적 키(authgate는 사람용, filegate 호출자는 서비스). 서비스가 filegate를 아는 것이지 반대가 아니다.

## 경계선

- **사용자 권한 검사는 영원히 서비스 몫.** filegate에 유저 개념은 어떤 형태로도 없다.
- **클라이언트 메타데이터는 불투명.** 저장·반환만, 해석·쿼리 금지.
- **관리 공간은 전부이거나 전무.** 외부 직접 접근은 데이터 파괴 (GC가 "기록에 없는 객체 = 쓰레기"를 전제).
- **채택은 opt-in.** 서비스 DB 인라인 데이터, CDN 자산, filegate 자신의 백업은 관할 밖.

## 결과

- 서비스에는 file_id만 남는다. 벤더 교체 = 설정 변경, 서비스 변경 0.
- filegate 장애 = 새 허가 불가일 뿐. 발급된 접근과 진행 중 전송은 계속 동작.

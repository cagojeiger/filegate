# spec 01: 선언형 설정과 부팅 검증

- Status: Draft
- Date: 2026-07-09
- 근거: ADR [000](../adr/000-identity.md), [001](../adr/001-multi-provider.md), [004](../adr/004-config-layers.md)

이 문서는 선언형 컨트롤 플레인이 **어떻게 연결되고 부팅에 검증되는지**만 정한다. 구체 필드·YAML 스키마·배치 전략은 구현과 서비스 통합이 확정한다 (이 문서 밖).

## 연결 구조

세 계층은 한 방향으로 참조한다. 소유자가 다르고, 참조 방향은 불변이다.

```text
clients.intents  ──가리킴──▶  storage_profiles  ──고름──▶  providers
(서비스 어휘)                  (운영자 카탈로그)            (물리 사실)
```

- intent는 provider를 직접 가리키지 않는다. 항상 profile을 경유한다 — 배치를 바꿔도 서비스 계약이 유지된다 (ADR 004).
- 모든 계층은 배포 설정(config)이다. 등록 테이블·admin 표면은 없다 (ADR 004).

## 배치 모드

profile은 배치 방식을 선언한다. 두 가지다:

- **auto**: 후보 provider 풀 중에서 전략으로 고른다 (기본 전략은 capacity 여유 후보 중 무작위 — ADR 001).
- **pin**: 특정 provider 하나에 고정한다.

구체 전략과 후보 풀 선정 규칙은 create 구현이 확정한다.

## 클라이언트 인증

- 선언형이다. 클라이언트는 config에 등록되고, 검증용 키 해시를 가진다 (raw 키 저장 안 함).
- 요청이 제시한 키를 해시해 등록된 해시와 대조하고, 맞으면 클라이언트 신원을 요청에 부착한다. 없으면 거부한다.
- 회전은 해시를 여럿 두고 config를 갱신·재배포하는 것이다.
- filegate 자체 키다. authgate에 의존하지 않는다 (공리 3).

헤더·해시 알고리즘·미들웨어 형태는 인증이 필요한 첫 오퍼레이션과 함께 구현한다.

## 부팅 검증 매핑

부팅 시 참조 그래프 전체를 해석한다.

- 모든 intent → profile 존재? → profile이 고르는 provider(들) 존재? → 그 provider 설정 유효? 하나라도 dangling이면 **부팅 실패** (ADR 004: 깨진 참조는 부팅 실패).
- 해석 결과는 런타임이 쓰는 resolved map이다: intent → 후보 provider 풀. 이 map의 첫 소비자는 create의 배치 결정이다.
- 검증은 요청 경로 밖, 부팅 1회다. 런타임에 config를 다시 읽지 않는다.

## 경계선

- 이 문서는 "무엇이 무엇을 가리키고, 무엇이 부팅을 막는가"만 정한다. 필드 목록·스키마·전략 세부는 정하지 않는다.
- 검증 매핑의 구현은 create와 함께 온다 — create가 이 map의 첫 소비자이기 때문이다.
- 범위 밖: quota 집행, tiering, 멀티 provider 배치 최적화 (다음 범위).
